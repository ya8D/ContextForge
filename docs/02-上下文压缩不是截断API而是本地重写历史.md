# 02 · 上下文压缩不是截断 API，而是本地重写历史

## 问题是什么

历史越滚越长，迟早两件坏事：模型对中段内容的召回率下降（"lost in the middle"），以及持续的
重放开销。所以需要「压缩」。但**压缩到底是在哪一层、对谁做的**？我一开始完全想歪了。

## 我原来怎么想

我以为压缩是**跟 API 打交道**的事——好像有个开关能告诉模型「你别看那么多历史了，帮我压一下」，
或者 API 那边维护着一份对话状态、我发个指令让它把旧的删掉。总之，压缩在我脑子里是一个
「对着远端的对话状态动手」的操作。

顺着这个错觉，我甚至去想「压缩失败了怎么办」「API 不支持压缩这段怎么办」——全是伪问题。

## 真相 / 原理

关键澄清：**API 是无状态的，它只认我们这一轮发过去的 `messages` 列表**。它不保存对话、没有
「那边的历史」可供压缩。所谓压缩，纯粹是**我们在本地把 `self.messages` 这个列表重写**——把中间
十几轮又臭又长的原文，替换成一条「前情摘要」消息。下一轮发过去的 `messages` 就变短了。API 根本
不知道发生过压缩，控制权 100% 在本地。

看 `compact_messages` 组装结果那一步就懂了——它返回的是一个**新的 Python list**：

```python
# context.py：compact_messages 结尾
summary_msg = {
    "role": "user",
    "content": f"{marker}\n{summary}",   # 一条摘要，顶替中段十几轮原文
}
new_messages = [head, summary_msg, *recent_messages]   # 头 + 摘要 + 最近几轮
return new_messages, stats
```

结构是**保头 + 压中段 + 保尾**：

```
[原始任务]  [第1轮]…[第N-k轮]         [最近 k 轮]
  ↑保留      ↑———— 压成一条摘要 ————↑    ↑—— 保留原文 ——↑
```

- **保头**：第一条（原始任务）必须留，否则模型忘了要干嘛。
- **压中段**：中间那些轮渲染成文本、调 LLM 总结成一条。
- **保尾**：最近几轮原文不动（正在进行的细节还有用）。

主循环下一轮 Think 时，`self.messages = new_messages`，发出去的就是这个短列表。**"压缩"发生在
本地内存里的一次 list 重写，跟 API 无关**。

> 这也解释了为什么压缩逻辑能被**零成本单测**：切分、保头保尾、组装全是纯 Python list 操作，
> 唯一的副作用（调 LLM 生成摘要）通过 `summarizer` 回调注入，测试传个假回调就能覆盖全部决策逻辑，
> 不烧一分钱。见 [05 · harness](./05-harness-用代码强制而非靠模型自律.md) 里讲的「注入式设计」。

## 学到的道理

- **"压缩上下文"是本地行为，不是 API 特性**。想通这一点，一大堆伪问题（"API 支不支持压缩"）
  自动消失，剩下的全是我能完全掌控的本地逻辑。
- **控制权在本地 = 可以随心所欲地客制化**。既然是我在重写 list，那"保什么、删什么、压多狠"
  全由我定——这直接催生了 [03 · 客制化压缩](./03-compact该客制化而非通用.md)。
- **无状态 API 是特性不是缺陷**。正因为它不替我存状态，我才对上下文有完全的控制。有状态的 API
  反而会把这份控制权收走。配合 [01 · KV Cache](./01-kv-cache-每轮全量历史为何不贵.md)，
  "无状态 + 每轮全量重发 + 本地重写压缩"这套组合，既可控又不贵。
- **搞不清一个机制在哪一层，就会凭空造出一堆不存在的难题**。先问「这事到底是谁在做、对谁做」，
  往往比直接想「怎么做」更重要。
