# 04 · compact 指令本身也要过模型的安全判断

## 问题是什么

[03](./03-compact该客制化而非通用.md) 让用户能给压缩下自然语言指令（directive）。给 T5-A 写
真实 e2e 测试时，我想验证"directive 说删什么就真删什么"，于是写了一条听起来很干脆的指令：
把带某个标记的内容**全部删除、绝不写进摘要**。

结果真实模型直接拒答：`Sorry, I can't assist with that.` 压缩没跑成，测试红了。

## 我原来怎么想

第一反应：**是不是题材敏感**？我压的是些技术日志、探针值，没什么违规内容啊。于是我怀疑是不是
某个词踩了雷，换了几个近义词试，还是被拒。我一度以为是模型抽风，或者这条 e2e 天生不稳定。

## 真相 / 原理

复盘发现，触发拒答的**不是题材，而是「意图的形状」**。我的 directive 长这样（示意）：

> "把所有带标记 `XXX` 的内容**全部删除、绝不写进摘要**，一条都不许出现。"

这个措辞的意图形状是——**系统化地删除/隐藏带特定标识的东西**。叠加上：

- 类 ID 的标记词（`XXX_1234` 这种，像某种记录编号）
- 系统日志的语境（压缩的就是一段 agent 运行历史）

三者合起来，在模型看来非常像**"清理/隐藏审计记录"**这类可疑操作——于是安全判断介入、拒答。
模型不是看懂了"这是无害的技术总结"，它看到的是"用户让我系统性地抹掉带标识的记录"。

**解法出奇简单**：把措辞从"删除/隐藏"的负向表述，换成"聚焦/取舍"的正向表述。语义等价，但意图
形状完全不同：

| 会触发拒答（负向、像清理记录） | 语义等价但不触发（正向、像做重点） |
|---|---|
| "把带 X 的内容全部删除、绝不出现" | "只聚焦保留和登录相关的报错，其余从简" |
| "隐藏所有 X 记录" | "摘要重点放在 Y，X 类噪声降频即可" |

在实测里也验证过同类现象：验证「口癖被压掉」时，断言不写成"抹净"，而是**"从 ≥6 次降到 ≤1 次"**
——用"降频/去重"的正向框架，既贴合压缩的真实语义（本就是降频而非抹除），也不触发安全判断。
（见 [tests/test_e2e.py](../tests/test_e2e.py) 里 directive 相关用例。）

## 学到的道理

- **用户给压缩下的 directive，本身也要过模型的安全判断**。压缩这一步是要真的调 LLM 的，
  你的指令就是它收到的 prompt——指令写得像"系统化删除/隐藏带标识的东西"，越强硬越可能被挡回。
- **触发拒答的是「意图形状」，不是关键词**。换近义词没用，因为模型判的是整体意图像不像可疑操作。
  要改的是**表述的框架**：负向的"删除/隐藏/不许出现" → 正向的"聚焦/取舍/降频"，语义不变、形状变了。
- **这是"客制化压缩"的一条真实边界**。给用户开自然语言指令的口子，就得知道这个口子连着模型的
  安全判断——不是所有措辞都能原样透传。设计客制化能力时，要把这层考虑进去（比如提示用户用正向表述）。
- **遇到"模型抽风"，先假设是自己触发了什么，而不是模型坏了**。这次要不是去想"我的指令在模型眼里
  长什么样"，就会一直归咎于"e2e 不稳定"，错过一个真实且可复现的机制。
