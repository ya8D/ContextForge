# 07 · sub-agent 本质是一个工具

## 问题是什么

有些子任务很啰嗦——要读一大堆文件、多步探索，但主 agent 其实只关心它的**结论**。如果让主 agent
自己去做，那一堆中间过程（读文件的全文、试错的弯路）会把主上下文塞满、冲淡真正重要的信息。

「sub-agent」就是来解决这个的：派个独立的 agent 去啃这个子任务，只把结论回传。但 sub-agent 在我
脑子里一度是个**很重、很特殊**的东西——好像需要专门的调度框架、消息总线、生命周期管理。

## 我原来怎么想

我以为 sub-agent 是一套**独立的子系统**：得有个"agent 管理器"负责创建、追踪、回收子 agent，
得设计主子之间的通信协议，得处理"上下文隔离"这种听起来很高级的机制……总之是个大工程。

尤其"上下文隔离"，我以为要专门写代码去**保证**两个 agent 的历史不互相污染。

## 真相 / 原理

其实 sub-agent 就是**一个普通的工具**——一个 `@tool` 装饰的函数，内部 `new` 一个 Agent、让它跑完、
返回结论。没有管理器，没有消息总线：

```python
# agent.py：spawn_subagent —— 就是一个 @tool
@tool({"task": "交给子 agent 独立完成的子任务描述，要自包含"})
def spawn_subagent(task: str) -> str:
    sub = Agent(
        tools=subagent_tool_schemas(),   # 受限工具集：剔除 spawn_subagent，防无限递归派生
        max_iterations=15,               # 子任务给更小的轮数上限，防它自己失控
    )
    result = sub.run(task)
    return f"[子 agent 完成] {result}"    # 只回传结论；sub.messages 留在它自己那里
```

**"何时派子 agent"的决策交给模型**——它就是工具菜单里的一项，模型判断某个子任务值得外包时就调它，
跟调 `read_file` 没有本质区别。**决策交给模型，执行和约束留给代码**。

而那个听起来很高级的"上下文隔离"，**根本不需要专门实现**——它是**两个 Agent 实例的 `messages`
天然是两个独立的 Python list** 这个事实**免费带来的**：

```
主 agent.messages = [...]     # 一个 list
sub  = Agent()
sub.messages      = [...]     # 另一个 list，和上面毫无关系
```

主 agent 调 `spawn_subagent`，子 agent 在自己的 `sub.messages` 里吭哧跑完整个 TAOR 循环，中间过程
全在子的 list 里；结束只 `return` 一个字符串。子的完整历史 `sub.messages` **压根没有回灌**主 agent。
隔离不是"实现"出来的，是"两个对象各有各的 list"这个再普通不过的事实**本来就成立**的。

> 顺带：`spawn_subagent` 为什么定义在 `agent.py` 而不是 `tools.py`？因为它要 `new Agent`。这背后
> 是一个关于依赖方向的教训，见 [08 · 循环 import](./08-循环import是东西放错了层的信号.md)。

## 学到的道理

- **sub-agent 不是子系统，是一个工具**。想通这点，一整套想象中的"调度框架"消失了——就是个函数，
  内部造个 Agent 跑一下。Agent 类本来就是可复用的，造第二个实例没什么特别。
- **"上下文隔离"是免费的，不用专门实现**。两个实例的 `messages` 是两个独立 list，天然隔离。
  我差点为一个"本来就成立"的性质去写一堆保障代码。**先问"这个性质是不是已经天然成立"**，
  再决定要不要动手实现。
- **决策归模型、执行归代码**——这条贯穿整个项目。子 agent 何时派由模型判断（它是工具），
  但"派生一个受限的、有轮数上限的子 agent"这个执行细节由代码写死（防递归、防失控）。
  跟 [05 · harness](./05-harness-用代码强制而非靠模型自律.md) 是同一个哲学。
- **越是听起来"高级"的词（隔离、调度、生命周期），越要警惕是不是自己把它想复杂了**。落到最朴素的
  实现（一个 list、一个函数），往往发现根本没那么玄。
