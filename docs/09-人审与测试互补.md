# 09 · 人审与测试互补

## 问题是什么

写了一堆 pytest、全绿，是不是就能安心了？我做这个项目时，**自动化测试全绿的同时，仍然藏着真
bug**——而且不止一个。这逼我想清楚：测试到底保证了什么、又保证不了什么？人肉 review 在其中是什么
角色？

## 我原来怎么想

朴素的想法：**测试绿 = 代码对**。测试是质量的护城河，跑绿了就说明没问题，人再去读代码是浪费时间。

这个想法在这个项目里被**两次实锤打脸**——两个真 bug 都是**逐行读代码**揪出来的，当时自动化测试
全绿、一片祥和。

## 真相 / 原理

### 人审揪出的：测试压根没覆盖的设计缺陷

逐行读 P4 的 `LoopDetector`，揪出两个真实缺陷（都成立，不是过虑）：只看第一个工具会**漏报+误报**、
命中后 reset 会**放水**（详见 [06](./06-死循环检测-reset反而放过真循环.md)）。这些是**设计层面**
的问题——测试当时全绿，因为**根本没有测试去覆盖"多工具乱序循环"或"连续不听劝的重复"这些场景**。
测试只能验证"我想到要验的东西"，想不到的场景它不会替你发现。

还有个更隐蔽的——T3 把源码搬进 `src/` 包时：

```python
# agent.py：定位项目根（.env / traces/ 在根，源码在 src/contextforge/ 下）
_HERE = Path(os.path.abspath(__file__)).parent.parent.parent   # 上跳三级到项目根
```

搬迁前 `_HERE = Path(__file__).parent` 只跳一级就够。搬进 `src/contextforge/` 后，深度变了，
`.env` / `traces/` 会定位到错误的层——**纯路径 bug，跑单测根本照不出来**（单测不碰真实磁盘、不读
`.env`）。这是"结构性搬迁"带出来的，只有人去想"物理路径变了几层"才发现。

### 但反过来，测试也揪出过人眼漏掉的：

别把结论走偏成"测试没用、只信人审"。`chmod -R 777` 的拦截就是**测试抓到的真 bug**：
`check_command_safety` 先对命令 `.lower()`，把 `-R` 变成了 `-r`，正则没匹配上、危险命令漏过。
这种"大小写归一化的副作用"人眼扫代码极容易滑过去，是**测试**把它钉出来的。

## 学到的道理

- **测试和人审是互补的，不是替代关系**。一句话概括各自的职责：
  - **测试保证「不退化」**——把"已知该对的行为"钉死，防止后续改动把它改坏。它守的是**回归**。
  - **人审发现「设计对不对」**——多工具乱序、reset 放水、搬迁后路径错层，这些是"要不要这么设计"
    的问题，测试想不到就覆盖不到。它守的是**盲区**。
- **"测试全绿"只等于"我想到要测的都对了"，不等于"代码对了"**。绿灯的边界就是你测试用例的边界。
  最危险的 bug 恰恰藏在"你没想到要测"的地方——那正是人审该去的地方。
- **但也别反过来只信人审**：`chmod` 那个 `.lower()` 副作用，人眼极易滑过、测试一跑就现形。
  两者各有各的盲区，**叠加**才够。
- **实践上**：改完代码，跑测试（守不退化）+ 自己逐行读一遍关键逻辑（找设计缺陷），两样都做。
  这个项目里两个真 bug 全靠后者揪出——如果当时"测试绿就收工"，它们就漏进去了。
