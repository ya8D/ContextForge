# 06 · 死循环检测：reset 反而放过真循环

## 问题是什么

模型有时会卡在同一个失败上——反复调同一个命令、同样的参数，十几轮都不换思路，白烧 token。需要
一个「死循环检测」：连续 N 轮完全相同的动作，就判定鬼打墙、注入"换个思路"的提示打断它。

这机制听起来简单，但两个细节我都先想错了：**① 检测粒度**、**② 命中后要不要清零**。

## 我原来怎么想

**关于粒度**：一轮可能调多个工具，我图省事，只取**第一个工具**做指纹来比较。

**关于清零**：命中死循环、打断一次后，很自然地想"重新计数"——把记录 reset 清零，重新攒。感觉
干净利落，符合直觉。

这两个决定，一个会**漏报+误报**，一个会**放水**。

## 真相 / 原理

### 粒度：只看第一个工具，两头都错

只取 `[0]` 做指纹，会同时犯两个方向的错：

- **漏报**（该报没报）：多工具乱序循环，比如 `[读A, 读B]` 和 `[读B, 读A]` 交替。只看 `[0]`，指纹
  在 A/B 间跳，判不出这其实是同一个循环。
- **误报**（不该报却报）：`[跑测试, 读日志]` → `[跑测试, 写报告]`，整轮明明在推进，但 `[0]` 全是
  "跑测试"，被误判成循环。

解法：**把一整轮所有工具排序后合成一个指纹**：

```python
# harness.py：record_round
def record_round(self, tool_use_blocks) -> None:
    fps = sorted(self._fingerprint(b.name, b.input) for b in tool_use_blocks)
    self._recent.append(str(fps))          # 整轮压成一个指纹
    if len(self._recent) > self.max_same:
        self._recent.pop(0)                # 滑动窗口
```

- **排序** → `[读A,读B]` 和 `[读B,读A]` 指纹相同（顺序不该影响"是否同一动作"），漏报没了。
- **整轮参与** → `[跑测试,读日志]` 和 `[跑测试,写报告]` 指纹不同，误报没了。
- 只看**请求**（工具名+参数）、不含执行结果——结果每次可能不同（含时间戳），不该妨碍"反复调同一
  命令"的判定。

### 清零：命中后 reset 等于放水

判定循环靠"最近 max_same 次全相同"：

```python
# harness.py：is_looping
def is_looping(self) -> bool:
    if len(self._recent) < self.max_same:
        return False
    return len(set(self._recent)) == 1     # 最近 N 次全一样 → set 里只剩 1 个
```

假设 `max_same=3`。如果命中后 reset 清零，会发生什么？模型**不听劝、继续重复**同一个动作时：

```
轮1 X, 轮2 X, 轮3 X → 命中！打断 + reset 清零
轮4 X, 轮5 X, 轮6 X → 又攒够3个才再命中 → 又 reset
轮7 X, 轮8 X, 轮9 X → 再命中……
```

**每 3 轮才拦一次**，中间 2 轮照样在空转烧钱。reset 把"连续重复"切成了一段段，等于给不听劝的循环
**放水**。正确做法是**命中后不 reset**：

```
轮1 X, 轮2 X, 轮3 X → 命中！打断（不清零）
轮4 X → 最近3轮[X,X,X]仍全同 → 再次命中！（一次不漏）
```

滑动窗口天然处理了"改了动作就放行"：模型一旦换个动作，窗口里混入不同指纹，`set` 长度 >1，
`is_looping` 自动返回 False，无需 reset。最坏情况还有 `max_iterations` 兜底。

## 学到的道理

- **"看第一个"是偷懒，代价是同时漏报和误报**。判断"是否重复"要看**完整的一轮**，还要排序消除
  顺序的干扰。降维（只取 `[0]`）省的那点事，换来的是两个方向的判错。
- **"命中后清零重新计数"听着合理，实则放水**。reset 把连续的循环切成段，让每段的前 max_same-1 轮
  白白空转。对付"不听劝的重复"，应该是**每一轮都重新判定、一次不漏**，而不是拦一次歇一会儿。
- **判据只看"请求"不看"结果"**。循环的本质是"反复发同一个请求"，执行结果每次不同（时间戳之类）
  不该干扰判定——想清楚"我到底在判什么"，才能选对指纹的组成。

  一推演就崩"**的例子。写下机制的逐轮行为、在纸上跑几步，比拍脑袋定方案靠谱得多。
