## T5 — 客制化 compact（可定制的上下文压缩 · A+B 都做）

> 起源：本项目实操中我们发现——通用 compact 是「哑」的（到阈值就机械总结，不管为了什么），
> 理想的压缩应该是**客制化**的（策略可配置、可按需定制），而非通用/写死一套。
> 这条既给 myagent 加功能（A），也把设计思想写成学习笔记（B）。

### A —— 给 myagent 加「客制化 compact」功能

**为什么**：P3 的压缩策略是**硬编码**的（阈值 500K、保头保尾压中段、固定 summarizer prompt）。
客制化 = 让压缩策略从「写死」变成「可注入的策略对象」，用户能按需定制压什么、留什么、怎么压。

**做什么**（把 `context.py` 的压缩从固定流程改成可配置）：
- **保留策略可配**：保留哪几维（目标 / 已做 / 发现 / 下一步）可定制；summarizer 的 prompt 可换。
- **裁剪策略可配**：哪些内容优先删（如「过程噪声 / 填充文本」优先删）。
- **压缩执行者可切换**：默认「自压」（`_summarize` 当前就是一次干净、无历史无工具的独立调用，
  不存在"当事人被污染"的问题）；可切换成「**派一个带工具的子 agent 去压**」——子 agent 的
  真正增量价值不是"隔离"（`_summarize` 本来就隔离），而是**给压缩这一步配上工具和多轮判断力**：
  能回读原始文件核实某段历史是否还站得住、能多轮推敲该留什么该删什么，而不是像单轮 `_summarize`
  那样盲总结一次、无法验证（复用 P5 的 sub-agent 机制）。
- 设计上沿用项目一贯的**注入**思路（同 P3 的 summarizer、P4 的 runner）：
  把「压缩策略」做成一个可传入的对象/回调，`compact_messages` 接受它，而非内部写死。

**验证**：能传入不同的压缩策略跑出不同结果；默认策略与 P3 行为一致（向后兼容）；
「子 agent 压缩」模式真调一次，确认子 agent 真的用工具核实过（如回读了某个原始文件），
而不是和 `_summarize` 一样凭空总结——这才是子 agent 模式该验证的增量价值。

### B —— 学习笔记：`docs/07-compact该客制化而非通用.md`

**为什么**：把「通用 compact 是哑的、客制化才对、子 agent 的增量价值在于工具+多轮判断力
而非隔离」这个认知沉淀下来。

**内容要点**（沿用 T4 的叙事结构：问题 → 原来怎么想 → 真相/原理 → 学到的道理）：
- 通用 compact 的局限：不知道「当前目的」，只会机械总结，通用即中立即「哑」。
- 客制化 compact：压缩时朝定制策略裁剪（狠删噪声、重点留目的相关内容）。
- **一个容易讲错的类比，要纠正**：`_summarize` 本来就是一次干净、无历史无工具的独立调用，
  不存在"当事人被污染、需要靠子 agent 代为自救"这回事。子 agent 的真正增量价值，是
  **给压缩这一步配上工具和多轮判断力**——能回读原始文件核实某个结论是否还成立、能多轮推敲
  该留什么该删什么，而不是像单轮 `_summarize` 那样盲总结一次、无法验证。
- **活案例（反面教材，另一层教训）**：本项目开发中，AI 曾陷入「填充 token」故障——上下文里
  累积的垃圾文本形成正反馈，主动劫持后续生成、让 agent 丧失功能。这生动证明：**上下文卫生
  （context hygiene）是 agent 可靠性的命门，「清理/压缩」不是可选项而是必需品**——陷入故障的
  当事人没有余地自查自纠，只能靠外部机制兜底（阈值触发的压缩、甚至整个开新会话）。这是独立于
  「压缩执行者该不该换成子 agent」的另一层教训，两者都值得写但不要混为一谈。

**验证**：`docs/07-...md` 标题一眼见结论；把填充故障写成活案例，人读有共鸣、AI 读懂设计哲学。
