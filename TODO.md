**为什么**：把这一路学到的**认知性经验**沉淀下来，**给人 & AI 都看**（不是给 AI 的运行记录、也不是排错手册）。

**形式**：主题式叙事，每篇讲透一个「学到了什么」。结构统一：
**问题是什么 → 我原来怎么想 → 真相 / 原理 → 学到的道理**。
读起来像学习心得，标题一眼见结论。

**篇目**（`docs/` 下）：
- `01-kv-cache-每轮全量历史为何不贵.md` —— in=2 的真相：input 只是未缓存新增量，其余命中缓存
- `02-上下文压缩不是截断API而是本地重写历史.md` —— API 无状态，压缩 = 我们在本地重写 messages
- `03-harness-用代码强制而非靠模型自律.md` —— 约束卡在「请求 → 执行」之间，不靠模型自觉
- `04-sub-agent-本质是一个工具.md` —— 决策交给模型、执行和隔离留给代码
- `05-死循环检测-reset反而放过真循环.md` —— 清零机制看似合理，实则纵容不听劝的循环
- `06-循环import是东西放错了层的信号.md` —— 延迟 import 是治标，根治是把东西挪到正确的层
- 末尾感悟（贯穿 05/06）：**人 review + 测试互补**——测试保证「不退化」，人审发现「设计对不对」
- **（补充选题，T5-A 实操中发现）`08-compact指令本身也要过模型安全判断.md`** ——
  写 T5-A 的 e2e"删除"测试时，directive 说"把带某标记的内容**全部删除、绝不写进摘要**"，
  真实模型直接拒答（`Sorry, I can't assist with that.`）。根因不是题材敏感，而是"系统化删除/
  隐藏带特定标识的东西"这个**意图形状**像"清理/隐藏记录"，叠加类 ID 的标记词 + 系统日志语境，
  触发安全拒答。启示：**用户给压缩下的 directive 本身也要过模型的安全判断**——"删除/不许出现"
  这类措辞越强硬，越可能被挡回；换成正向的"聚焦/取舍"表述语义等价却不触发。这是"客制化
  compact"的一个真实边界，值得单独成篇（或并入 07）。

**验证**：`docs/` 下 6 篇 + 每篇标题一眼见结论；人读顺畅、AI 读能懂设计哲学。


## T5-B — 学习笔记：`docs/07-compact该客制化而非通用.md`

> T5 的功能部分（A）已完成（见上方「已完成」区）。这里只剩把设计思想写成学习笔记（B），
> 和 T4 的六篇一起写、一起验证。

**为什么**：把「通用 compact 是哑的、客制化才对、子 agent 的增量价值在于工具+多轮判断力
而非隔离」这个认知沉淀下来。

**内容要点**（沿用 T4 的叙事结构：问题 → 原来怎么想 → 真相/原理 → 学到的道理）：
- 通用 compact 的局限：不知道「当前目的」，只会机械总结，通用即中立即「哑」。
- 客制化 compact：压缩时朝定制策略裁剪（狠删噪声、重点留目的相关内容）。
- **一个容易讲错的类比，要纠正**：`_summarize` 本来就是一次干净、无历史无工具的独立调用，
  不存在"当事人被污染、需要靠子 agent 代为自救"这回事。子 agent 的真正增量价值，是
  **给压缩这一步配上工具和多轮判断力**——能回读原始文件核实某个结论是否还成立、能多轮推敲
  该留什么该删什么，而不是像单轮 `_summarize` 那样盲总结一次、无法验证。
  （★ T5-A 实操已实证这一点：子 agent 执行者真的 read_file 回读核实了探针值，摘要里写
  "已回读…核实，当前仍成立"——写笔记时可直接引用这个活案例。）
- **活案例（反面教材，另一层教训）**：本项目开发中，AI 曾陷入「填充 token」故障——上下文里
  累积的垃圾文本形成正反馈，劫持后续生成、让 agent 丧失功能。这生动证明：**上下文卫生
  （context hygiene）是 agent 可靠性的命门，「清理/压缩」不是可选项而是必需品**——陷入故障的
  当事人没有余地自查自纠，只能靠外部机制兜底（阈值触发的压缩、甚至整个开新会话）。这是独立于
  「压缩执行者该不该换成子 agent」的另一层教训，两者都值得写但不要混为一谈。

**验证**：`docs/07-...md` 标题一眼见结论；把填充故障写成活案例，人读有共鸣、AI 读懂设计哲学。
