# 经典案例：指令驱动压缩（百家姓）

演示 `/compact <要求>` 在**短会话**里按用户要求压缩历史，且压缩后模型正确地只用压缩后的数据回答、
不补全被删内容。原始 trace：`traces/run_20260709_171737`（本目录是其精选留档）。

## 场景

1. **task_01**：用户让 agent「列出中国姓氏最多的前一百名，存进临时上下文，禁止建文件」。
   模型答出前100名（`messages` = `[A问题, B回答]` 两条）。
2. **task_02/turn_01**：用户敲 `/compact 删除后40位`。
   - 这条 `/compact` 指令被 CLI 拦截、**不进历史**，直接触发指令驱动压缩。
   - `compact_by_directive` 把 `[A, B]` 压成 `[A, 摘要]`：保住原始任务 A（锚点），其余按要求压成
     一条摘要（只留前40名）。
   - 看 `task_02/turn_01.json` 的 `messages_sent` 三条：
     - `[0]` 原始任务（保头 A）
     - `[1]` `[前情摘要 · …按要求：删除后40位]` + 护栏（禁止补全）+ 前40名 —— **这就是压缩后的历史**
     - `[2]` 「现在上下文里排第一的姓氏是谁？只回答这一个问题」
   - 模型回复：**「王」**（就一个字）——只用了压缩后的数据，没有补全被删的 41-100 名。

## 为什么值得留存

- **一图看懂指令驱动压缩**：`messages` 真被本地重写成 `[头, 摘要]`，API 无从知晓，控制权全在本地。
- **护栏 + 精确提问 = 压缩生效**：摘要里带「禁止补全」护栏；配合明确的提问，模型规矩地只用压缩后内容。
- **反衬一个测试教训**：若这一步问的是空泛的 `HI`，模型会把它当「新开场」、重答最初的「列前一百」，
  造成「压缩没生效」的假象。验证压缩效果要用**明确只需压缩后数据**的问题。
