## 一、四个计费维度的定义

你在 agent 日志里看到的 `in=899 (cache_read=0, cache_write=0) / out=77`，对应下面 4 个维度：

| 维度 | Anthropic 字段 | 含义 | 计费特点 |
|---|---|---|---|
| **in（输入）** | `input_tokens` | 这轮发出、**未命中缓存、首次计算**的 token | 全价（基准） |
| **cache_write（写缓存）** | `cache_creation_input_tokens` | 首次把前缀**写入**缓存供后续复用 | Anthropic ≈1.25× 基准 |
| **cache_read（读缓存）** | `cache_read_input_tokens` | 命中缓存、**从缓存读**的 token | Anthropic ≈0.1× 基准（省 90%） |
| **out（输出）** | `output_tokens` | 模型这轮**生成**的 token | 输出价（通常 4~5× 输入） |

> **DeepSeek 的口径不同**：它直接给 `cache hit`（≈读缓存）和 `cache miss`（≈首次/写）两档输入价，
> 没有单独的「写缓存溢价」。下表把两家对齐到同 4 个维度便于对比。

再看之前 turn_02 那轮（`in=2, cache_write=1958, out=140`），体会缓存的影响：

- 注意：**写缓存那 1958 token 花了 $0.0122**，是这轮大头 —— 首次建缓存要付 1.25× 溢价；
  但只要后续轮次命中 `cache_read`（0.1×），马上就回本。这就是「保持前缀稳定」省钱的原理。

## 五、关键结论

1. **`input_tokens` ≠ 发出去的总量**：它只是「按全价算的部分」。真实发出量 = input + cache_write + cache_read。
2. **三种输入 token 单价差极大**：`cache_read`(0.1×) ≪ `input`(1×) < `cache_write`(1.25×)。多轮循环命中 `cache_read` 是省钱核心。
3. **最强对最强**：DeepSeek-R 比 Opus 4.8 便宜约一个数量级（输入/输出 ~9-11×，读缓存 ~3.6×）。
4. **价值权衡**：便宜不等于划算 —— 要结合任务难度、成功率、agent 循环轮数综合看（贵模型少走几轮、少返工，总成本可能反而低）。这正是第 18 章「成本控制」和「模型路由」要解决的问题。
